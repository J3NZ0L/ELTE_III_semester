A fájlrendszer fogalma

Az NTFS fájlrendszer

Az NTFS fájlrendszer (avagy New Technology File System, magyarul az új technológiájú fájlrendszer) a Microsoft korábbi FAT fájlrendszerét váltotta le, melyet az MS-DOS és az azt követő Microsoft Windows operációs rendszerek esetén használtak, egészen a Microsoft Windows NT operációs rendszer megjelenéséig.

A Microsoft Windows NT operációs rendszerben jelent meg először az NTFS fájlrendszer, majd a későbbi Microsoft Windows verzióknak (Microsoft Windows 2000, Microsoft Windows XP, Microsoft Windows Server 2003, Microsoft Windows Vista, Microsoft Windows Server 2008, Microsoft Windows Server R2, Microsoft Windows 7) az NTFS fájlrendszer lett a szabványos fájlrendszere.

A korábbi Microsoft Windows operációs rendszerek – Microsoft Windows 95, Microsoft Windows 98, Microsoft Windows 98 Second Edition és Microsoft Windows Millennium Edition – nem képesek natív módon olvasni az NTFS fájlrendszert, tehát natívan nem támogatják az NTFS fájlrendszert, bár léteznek programok erre a célra is.

Manapság széles körben elterjedt az NTFS fájlrendszer, ugyanis rengeteg újítás, újdonság és extra funkció érhető el benne, ami az elavult FAT fájlrendszerben nem volt elérhető, és emiatt a FAT fájlrendszer hiányosságaként soroltuk fel az előző pontban. Ilyen például a metaadatok támogatása, a fejlettebb adatstruktúrák támogatása a sebesség, a megbízhatóság és a lemezterület-felhasználás érdekében, valamint már rendelkezik hozzáférés-védelmi listával, és megtalálható benne a naplózás is.

Kérdés merülhet fel az olvasóban: ha ennyivel jobb az NTFS fájlrendszer, akkor miért használjuk napjainkban a FAT fájlrendszert is, azon túl, hogy az minden operációs rendszerrel kompatibilis? Erre a kérdésre a választ már maga a kérdés megadja, hiszen az NTFS fájlrendszer legfőbb hátránya, hogy a nem Microsoft operációs rendszerek esetében meglehetősen korlátozott a támogatottsága. Ez a korlátozott támogatottság arra vezethető vissza, hogy az NTFS fájlrendszernek a pontos specifikációja a Microsoft szabadalma.

Az NTFS fájlrendszernek három verziója létezik:

  • v1.2 – Microsoft Windows NT 3.51, Microsoft Windows NT 4
  • v3.0 – Microsoft Windows 2000
  • v3.1 – Microsoft Windows XP, Microsoft Windows Server 2003, Microsoft Windows Vista, Microsoft Windows Server 2008, Windows Server 2008 R2, Microsoft Windows 7

Előfordul, hogy ezekre a verziókra v4.0-ként, v5.0-ként és v5.1-ként is hivatkoznak abból kifolyólag, hogy miképpen változott a Microsoft Windows operációs rendszer verziószáma.

Az egymást követő operációs rendszerekben, ahol változott az NTFS fájlrendszer egy újabb verzióra, ott újabb funkciókat is vezettek be. Ilyen például a Microsoft Windows 2000 operációs rendszer, amelynek a fájlrendszerében (NTFS v3.0) jelent meg először a kvótatámogatás.

Az NTFS és a FAT fájlrendszer különbségei

Eredetileg a FAT fájlrendszer -struktúra úgy definiálja a maximálisan létrehozható klasztermennyiséget, hogy maximálisan 65517 klaszter helyezkedhet el egy FAT16 fájlrendszer -partíción (FAT16 köteten). Emiatt ha nagyobb merevlemezt vásárolunk, akkor a formázó programnak muszáj megnövelnie a klaszterméretet, hogy az egész merevlemezt egy kötetként formázhassuk meg. Emiatt nem célszerű nagyméretű merevlemezek megformázásakor a FAT fájlrendszert kiválasztani, mivel nagyon pazarló. Miért is tekintjük a FAT fájlrendszert rendkívül pazarlónak ebből a szemszögből?

Mivel a nagyméretű merevlemezek esetén a fentebb említett ok miatt nő a klaszterméret, így a kisméretű fájlok tárolása esetén a fájl a saját valódi méreténél jóval nagyobb helyet foglal el a merevlemezen, pontosan egy klaszterméretnyi mennyiséget, mivel – mint tudjuk – a klaszter a legkisebb kezelhető egység a merevlemezen. Így a sok kisméretű fájl tárolásakor feleslegesen foglalnak a fájlrendszer „hibája” miatt a valódi méretüknél lényegesen nagyobb helyet a lemezen, emiatt pazarló a FAT fájlrendszer.

Az NTFS fájlrendszer 64 bites indexeket használ a klaszterek kiválasztásához, így nem pazarol az amúgy is szűkös hellyel. A 64 bites indexelési módszernek köszönhetően az NTFS fájlrendszernek az elvileg elérhető maximális partíciómérete 16 exabájt (16 EB), ám a gyakorlatban 2 terabájtnál (2 TB) nagyobb partíciókat már nem kezel.

Az NTFS fájlrendszer teljesen a Microsoft Windows NT operációs rendszer biztonsági modelljére épül. Pontosabban a DACL (Discretionary Access Control List) és SACL (System Access Control List, magyarul hozzáférési listák) biztonsági megfontolásaira épül, így az NTFS fájlrendszeren belül megadható, hogy melyik felhasználó mikor és mit végezhet egy-egy fájllal, továbbá minden elvégzett művelet visszakövethető benne. Ez a biztonsági modell az, amivel az NTFS fájlrendszer valóban hatalmas előrelépés volt a FAT fájlrendszerhez képest. Az NTFS fájlrendszer mellett szól még az is, hogy egy többfelhasználós környezetben, mint amilyen a Microsoft Windows NT operációs rendszer szerver verziója, feltétlenül szükséges egy ilyenfajta biztonsági modell követése.

A FAT fájlrendszerrel ellentétben az NTFS fájlrendszer volt az első olyan Microsoft Windows platformú fájlrendszer, ami natívan (bármiféle „hack” nélkül) támogatta a hosszú fájlneveket.

A FAT fájlrendszer legnagyobb hibáját, miszerint a FAT fájlrendszerben nincs semmilyen hibatűrő megoldás, az NTFS fájlrendszer ügyesen megoldotta.

A hibatűrő megoldás alatt azt értjük, hogy ha a nyitott fájlok módosítása, szerkesztése során fellép egy esetleges rendszerhiba, például a számítógép „lefagy”, akkor a fájlrendszer -partíción tárolt adatok nem vállnak inkonzisztenssé.

Az NTFS fájlrendszer megoldása az lett, hogy vezet egy beépített fájlművelet-végzési nyilvántartást (azaz egy úgynevezett log fájlt). Ebbe a speciális log fájlba minden egyes fájlművelet során egy bejegyzést tesz. Így amikor egy esetleges rendszerhiba következik be (pl. a fentebb említett „lefagyás”), akkor az újrabootolás során a log fájlok alapján azonnal meg lehet állapítani, hogy a rendszerhiba bekövetkezésekor történt-e bármiféle adatvesztés, vagy sem. Amennyiben a log fájlok alapján megállapítható az adatvesztés megtörténte (a log fájlban benne lesz, hogy a fájlmódosító művelet félbemaradt), akkor el kell indítani az úgynevezett CHKDSK (lemezellenőrző) programot. A CHKDSK program a log fájlt felhasználva megpróbálja visszaállítani az eredeti konzisztens állapotot. A log fájl minden bejegyzése két típusú lehet: vagy ismétlő (redo), vagy visszavonó (undo) parancsot tartalmazhat.

A log fájl mérete a merevlemeztől függően két és négy megabájt között (2 MB – 4 MB) mozoghat. Ha az NTFS fájlrendszer nem vizsgálná meg mindig, hogy az ismétlő (redo) és a visszavonó (undo) információk szükségesek-e hibajavítás esetére, akkor a log fájl mérete gyorsan megnőne, és így hamar be is telne. Ennek az elkerülése végett az összes módosított adatot lemezre menti az NTFS fájlrendszer, és törli a log fájl tartalmát. Ez a vizsgálat az úgynevezett checkpoint- (határpont-) képzés. A vizsgálat körülbelül minden 4. vagy 5. másodpercben megtörténik.

Példának okáért tegyük fel, hogy éppenséggel az NTFS fájlrendszerünkben egy fájl törlése közben egyszerűen „lefagy” a számítógépünk. Természetesen nem esünk kétségbe, mivel tudjuk, hogy a fájlrendszerünk a FAT fájlrendszerrel ellentétben naplózást készít egy log fájlba. Újraindítjuk a számítógépünket, a bootolás közben, a visszaállítás során kiderül, hogy az adott fájl nem lett törölve, a módosítás befejezve, így a művelet újra végrehajtandó. A másik esetben, azaz a visszavonás esetében például egy fájlhoz fűzünk hozzá egy másik fájlt, és a fájl méretének, illetve a fájl hozzáférési dátumának beállításánál „fagy le” számítógépünk. Ekkor sem esünk kétségbe az NTFS fájlrendszer használatakor, ugyanis a hibajavítás az eredeti állapot visszaállításából fog állni: az eredeti fájlméret és hozzáférési dátum megadása.

Egy NTFS fájlrendszerbeli partíció létrehozása

Egy partíció formázásakor és a fájlrendszer beállításakor egy úgynevezett Master Boot Record (MBR) jön létre a háttértárolón. A Master Boot Record, vagy más néven a partíciós szektor a merevlemez legelső szektorának (azaz az első lemezfelület első sávjának, első szektorának) elnevezése. Csak a particionált merevlemezeknek van Master Boot Recordjuk. A Master Boot Record a merevlemez legelején, az első partíció előtt található meg. A Master Boot Record egy kisebb végrehajtható kódot, amit Master Boot Code-nak hívunk, és egy partíciós táblát tartalmaz. Egy új partíció becsatolásakor a Master Boot Record végrehajtja a Master Boot Code-ot, és átadja a vezérlést a lemez boot szektorának.

Elemek

Leírás

Hard disk

Háttértároló, ami egy vagy két partíciót tartalmaz.

Boot sector

Bootolható partíció, amely információkat tartalmaz a lemez kialakításáról, a fájlrendszer struktúrájáról. Itt tárolódik a boot code is, amely az Ntdlr-t tölti be.

Master Boot Record

Olyan végrehajtható kódot tartalmaz, amit a rendszer BIOS tölt be a memóriába. A kód a partíciós tábla alapján eldönti, hogy melyik partíció aktív vagy bootolható.

Ntldlr.dll

A processzort (CPU) védett módba állítja, elindítja a fájlrendszert, majd kiolvassa a Boot.ini fájl tartalmát. Ezek az információk meghatározzák a kezdeti beállításokat és a kezdeti „boot menu” opciókat.

Az Ntldr a Windows NT alapú operációs rendszerek betöltőprogramja, amelynek feladata az alapvető hardverkonfigurációs jellemzők felderítése, és az ezeknek megfelelő rendszermagváltozat betöltése és elindítása.

Ntfs.sys

System file driver az NTFS fájlrendszerhez.

Ntoskrnl.exe

Információt ad arról, hogy melyik „system device drivert” (eszközkezelő alkalmazás) kell betölteni és hogy mi a megfelelő betöltési sorrend.

Kernel mode

Az a végrehajtó mód, amely lehetőséget biztosít a kódoknak, hogy közvetlen hozzáférjenek az összes hardverhez és a memóriához a rendszerben.

User mode

Az a végrehajtó mód, ahol az alkalmazások futnak.
